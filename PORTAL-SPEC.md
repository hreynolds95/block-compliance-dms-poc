## 7.1 Overview

The staff-facing portal is a GitHub Pages site that provides:

- **Read-only document library** — all published In-Scope Documents, searchable and filterable
- **Compliance metrics dashboard** — review health, ownership accountability, tier/domain coverage
- **Change request initiation** — Document Submitters and Owners trigger the annual review workflow directly from the portal
- **New document request form** — any Block employee submits a self-service request; routed to Policy Team for conflict check and validity assessment (PoP 2.7.3)

The portal has **no document content editing capability**. All changes flow through the GitHub PR workflow defined in Section 4. The portal is the read interface; the repo is the write interface.

## 7.5 Change Request Initiation

Document Submitters and Owners initiate scheduled reviews via `annual-review.yml`. In Phase 1, this is a manual GitHub Actions dispatch. Phase 2 surfaces it as a portal button.

### Phase 1 — Manual dispatch

```
Document Owner / Submitter:
  1. Navigate to GitHub Actions tab in backend repo
  2. Select "Initiate Annual Review" workflow
  3. Click "Run workflow"
  4. Enter: doc_id (e.g. FC-017), change_type (immaterial/material), notes
  5. Workflow creates review branch and opens PR
  6. Portal data reflects status change on next data refresh
```

### Phase 2 — Portal-triggered dispatch

```
Authenticated Document Submitter / Owner:
  1. Finds document in policy library
  2. Clicks "Request Review" button on document row (visible only to Submitters/Owners)
  3. Portal modal: select change_type, add notes
  4. Portal calls GitHub API:
       POST /repos/{owner}/{repo}/actions/workflows/annual-review.yml/dispatches
       Authorization: Bearer {user_token}
  5. Workflow fires; PR opened; portal reflects status on next refresh
```

**Off-cycle material changes** are not initiated via the portal — these are opened directly as PRs by the Document Submitter in the backend repo. The portal surfaces the resulting `in-review` status once the PR is opened.

## 7.6 New Document Request Form

Any Block employee can submit a new document request. The request is filed as a GitHub Issue using the `new-document-request.yml` template. The Policy Team evaluates conflict/overlap and approves or rejects (PoP 2.7.3).

**Issue template:** `.github/ISSUE_TEMPLATE/new-document-request.yml`

**Required fields:**

| Field | Description |
|---|---|
| Proposed title | Working title for the new document |
| Document type | Policy / Standard / Procedure |
| Proposed tier | 1 / 2 / 3 / Unsure |
| Compliance domain | Governance, Financial Crimes, etc. |
| Business / entity scope | Which brands or legal entities this document covers |
| Rationale | Why this document is needed; what requirement or gap it addresses |
| Conflict check attestation | Checkbox confirming the requester has searched existing documents and believes no overlap exists |
| Potentially overlapping documents | List of existing doc IDs or titles that may be related |
| Proposed document owner | Name and title of the proposed Document Owner |
| Implementation plan link | External link only (Google Drive, etc.) — per DocArchitect constraint, implementation plans are not stored in GitHub |

**Processing workflow:**

```
Any employee opens new-document-request Issue
  │
  ├─ Issue auto-labelled: new-doc-request
  │
  ├─ Phase 2: Goose-assisted semantic search
  │     Appends list of potentially overlapping existing docs to Issue
  │
  ├─ Policy Team reviews (PoP 2.7.3):
  │     - Conflict check: does this duplicate / overlap existing docs?
  │     - Scope review: does this fit the governance model and taxonomy?
  │     - Tier recommendation
  │
  └─ Decision:
       ├─ Approved → Policy Team assigns doc_id, creates registry entry,
       │              scaffolds branch with correct template
       └─ Rejected → closes Issue with reasoning and reference to
                     existing document(s) that cover the need
```

> **Phase 1:** Policy Team review and branch scaffolding are performed manually by `@hreynolds95`. Phase 2 adds automated conflict search and registry scaffolding.

## 7.7 Integration Points

| Portal feature | Backend dependency | Integration mechanism |
|---|---|---|
| Policy library table | `site/docs-data.json` | Generated from `docs/**/*.md` frontmatter by `generate_docs_data.py` on each refresh |

| Document detail — LogicGate link | `logicgate_record_id` frontmatter field | Constructed URL: `https://block.logicgate.com/records/{id}` |

| Change request initiation | `annual-review.yml` | Phase 1: manual dispatch; Phase 2: GitHub API call |
| New document request | `.github/ISSUE_TEMPLATE/new-document-request.yml` | GitHub Issue form in backend repo |
| Exception register (Phase 2) | `exceptions/exceptions.jsonl` | Read at runtime; rendered client-side |
